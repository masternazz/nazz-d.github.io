Operations & Alerting Stack

Centralized alerting for the homelab using mailcow as the internal notification path. The goal was simple: important service failures should reach an inbox, not sit quietly inside separate dashboards.

mailcow OPNsense Monit Wazuh Uptime Kuma Proxmox Authentik

What I Built

  • Configured mailcow to accept trusted internal alert traffic from the lab network.
  • Installed the wildcard certificate used by the lab so SMTP STARTTLS presented a trusted certificate instead of a self-signed one.
  • Enabled notification paths for Proxmox, Wazuh, Uptime Kuma, Authentik, OPNsense Monit, and the media management stack.
  • Documented per-service rollback steps so a bad alert change could be reversed without guessing.
  • Kept sensitive implementation details out of public documentation while preserving the operational story.

Skills Demonstrated

  • Linux service configuration and log validation
  • SMTP relay behavior, STARTTLS, and certificate troubleshooting
  • OPNsense Monit configuration and firewall-aware alerting
  • Wazuh and Uptime Kuma notification integration
  • Operational documentation, change tracking, and rollback planning
  • Security judgment around what should and should not be published

Alert Flow

How alerts move from service events to a place where they can be acted on.

Service Event Firewall, virtualization, identity, monitor, or app detects a problem

Each service keeps its own local logs and alert trigger conditions.

Internal Relay Trusted lab service sends SMTP to mailcow

Internal systems use a controlled path instead of external free-form mail settings.

Delivery mailcow routes the notification to the admin mailbox

STARTTLS uses the lab wildcard certificate so the path is clean and verifiable.

Response Runbook notes explain the change, validation, and rollback

The same notes that describe the alert also describe how to undo or adjust it.

Monitoring

Uptime + Security

Uptime Kuma covers service reachability while Wazuh handles security-focused alerts. The two systems answer different questions: is it reachable, and did something suspicious happen?

Firewall

OPNsense Monit

Monit gives the firewall its own alerting path for service-level issues. That matters because the firewall is the dependency many other alerts rely on.

Recovery

Rollback Notes

Every major service change includes a rollback note: which config changed, what to remove, and what to restart. That turns troubleshooting into a controlled process.

Implementation Notes

The useful part of this project was not just "turning on notifications." Each service had a different configuration surface: some used config files, some used APIs, some stored settings in SQLite, and some required service restarts. The work was validating each path end to end, checking logs, confirming delivery, and writing down exactly what changed.

I also kept public-facing documentation sanitized. Internal IPs, account details, relay trust rules, and mailbox-specific settings do not need to be public for the project to show the work.
