Homelab MCP Server

Built a custom Model Context Protocol server running on a dedicated Proxmox LXC. It exposes 55 validated tools across the homelab so MCP-compatible clients can run approved checks and service actions over SSH.

Node.js MCP SDK OPNsense API Proxmox Wazuh Claude Code Gemini CLI stdio / SSH

What It Does

  • Runs on a dedicated Debian 12 LXC (64 MB RAM idle) connected to the management VLAN and reached over SSH.
  • Exposes 55 structured tools across AdGuard, Proxmox, OPNsense, NPMplus, Mailcow, Wazuh, Uptime Kuma, Authentik, Headscale, Jellyfin, and Home Assistant.
  • MCP-compatible clients — Claude Code, Gemini CLI, VS Code Copilot, Cursor — connect via stdio transport over SSH and call tools natively.
  • Every tool is a typed, validated function that returns a structured result — no copy-pasting or manual API formatting.
  • Credentials stay on the server in a local .env file — never passed over the MCP connection itself.

Architecture

  • Transport: stdio over SSH — the AI client SSHes into the LXC and pipes MCP protocol through the connection. No exposed HTTP port, no public auth surface.
  • Runtime: Node.js 20 LTS, @modelcontextprotocol/sdk for the server framework, zod for input validation, node-fetch for API calls.
  • Structure: index.js loads .env, imports all tool modules, registers each tool with the MCP server, starts stdio transport.
  • Each module handles its own auth (Basic, Bearer, cookie, API key) and exposes named handler functions — adguard.js, opnsense.js, proxmox.js, npmplus.js, mailcow.js, monitoring.js, services.js.
  • Proxmox access: SSH into firstmoon and use pct exec — no Proxmox API token needed for LXC command execution.

Transport

stdio over SSH

The MCP server runs over stdio transport — the AI client SSHes into the LXC and pipes the MCP protocol through the connection. No HTTP port exposed to the network, no additional authentication surface. The SSH key is the only credential needed to connect.

Multi-client

Works with Any MCP Client

The same server config works with Claude Code, Gemini CLI, VS Code Copilot, and Cursor. Each client SSHes in and speaks the MCP protocol — the server doesn't care which client it's talking to. Add the server once, use it from any tool.

Integration

Auth Per Service

Each tool module handles its own authentication independently — AdGuard uses Basic auth, OPNsense uses API key/secret, NPMplus uses a session cookie, Mailcow uses an API key header, Proxmox goes over SSH. Credentials are loaded from a local .env at startup and never touch the MCP connection.

Real Infrastructure Tasks Completed With It

Infrastructure tasks completed through controlled MCP tool calls.

NIC Swap Recovery

Hardware swap broke all VLANs after a driver rename changed the firewall interface names. Used OPNsense tools to migrate VLAN parents and correct DHCP ranges.

Proxy Host Audit

Listed all 28 NPMplus hosts, tested every backend, diagnosed two failures (wrong path and firewall block), and corrected both through tool calls in a single session.

Firewall Debugging

Diagnosed a 502 on webmail by listing firewall rules, finding the RFC1918 block firing before the pass rule, and adding a correctly-sequenced pass rule via the OPNsense API.

DNS + Service Ops

Added DNS rewrites, checked DHCP leases, restarted services, queried Wazuh alerts, and checked Uptime Kuma monitor status through the MCP server.

How It All Connects

AI Client (Claude Code / Gemini CLI / VS Code / Cursor)
    │  SSH connection → MCP LXC
    │  stdio transport (MCP protocol)
    ▼
MCP Server (Node.js, LXC on Management VLAN)
    ├─ adguard.js    → HTTP Basic auth  → AdGuard Home
    ├─ proxmox.js    → SSH + pct exec   → Proxmox firstmoon
    ├─ opnsense.js   → HTTPS API key    → OPNsense
    ├─ npmplus.js    → HTTPS cookie     → NPMplus
    ├─ mailcow.js    → HTTPS API key    → Mailcow
    ├─ monitoring.js → HTTP / HTTPS     → Uptime Kuma, Wazuh
    └─ services.js   → HTTPS tokens     → Authentik, Jellyfin, HA, Headscale

Employer-Relevant Skills

This server runs on infrastructure I use for homelab management. I identified a workflow problem across multiple service APIs, designed an MCP-based solution, implemented it in Node.js with separate authentication handling for each service, and deployed it on dedicated LXC infrastructure. The issues it surfaced — cookie auth versus Bearer tokens, GET body rejection, and firewall rule ordering — are the same kind of integration friction that shows up in IT work.

What Building It Taught Me

Every service had different auth — AdGuard uses Basic, OPNsense uses an API key pair, NPMplus uses a session cookie it refreshes on login, Mailcow uses a header token. Getting all of them working in the same Node.js process without any leaking credentials across the MCP connection took some care. The NPMplus cookie approach in particular was annoying — it rejects GET requests with a body, which isn't obvious from the docs. Figuring out that the firewall rule sequence number matters (not just whether a rule exists) came from the same kind of trial and error you'd hit on any real network.
