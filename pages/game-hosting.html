Game Hosting & Discord Automation

12 self-hosted game servers across Minecraft and Steam titles, running on Pelican Panel and Wings inside Proxmox LXCs. Built a Discord bot from scratch that tracks live player counts across multiple protocols, auto-stops idle servers with a 10-minute warning, and exposes a Flask status API consumed by a custom public status site and Uptime Kuma monitors.

Pelican Panel Docker Python Discord API RCON Protocol Steam A2S Flask Proxmox LXC systemd

Game Server Infrastructure

  • Pelican Panel (open-source successor to Pterodactyl) on LXC 104, with a separate Wings daemon on LXC 121 — both on Proxmox firstmoon.
  • Wings runs each game server as an isolated Docker container with its own resource limits, filesystem, and network namespace.
  • 12 servers across two categories — Minecraft: Vanilla QOL (Fabric 1.21), Casket of Reveries (Forge 1.20.1), Supernatural (Forge 1.20.1), and two additional private Fabric instances. Steam: Abiotic Factor, Terraria (TShock), Subnautica co-op (×2), Satisfactory, LMP (KSP Multiplayer), and a custom multiplayer title.
  • Each server has a dedicated subdomain. Minecraft servers have SRV DNS records so players connect without specifying a port.
  • Modpack files (Modrinth, Prism, CurseForge formats) hosted on self-hosted Nextcloud.

Discord Bot — Built From Scratch

  • Posts a live status embed to a Discord channel, auto-updating every 60 seconds with player counts for all 12 servers, split into Minecraft and Steam sections with per-category online counts.
  • Player counts use different protocols per server: custom RCON client (pure Python) for Minecraft, Steam A2S UDP query for Abiotic Factor, TShock REST API for Terraria. Servers with no player API show — and are never auto-stopped.
  • Auto-stop logic: any server idle for 20 minutes is stopped automatically via the Pelican Wings API. Warning fires at 10 minutes. Servers with no player API are exempt.
  • Slash commands: /start, /stop, /servers — all use autocomplete dropdowns so there's no free-text server name to mistype. Bot presence updates live with active player count.
  • Deployed on a dedicated Debian LXC 122 (thirdmoon) as a systemd service — starts on boot, auto-restarts on failure.
Protocol

Multi-Protocol Player Queries

Each game server exposes player counts differently. Minecraft servers use RCON — I implemented the binary TCP wire protocol from scratch in Python: packet framing, struct packing, auth handshake, response parsing. Abiotic Factor uses the Steam A2S_INFO UDP query protocol including the challenge handshake. Terraria (TShock) has a REST API. Servers with no accessible player API return None — the embed shows — and auto-stop is skipped entirely for those servers.

Automation

Idle Auto-Stop

Any server sitting at 0 players for 20 minutes is automatically stopped via the Pelican Wings API — no manual intervention needed. A warning fires at 10 minutes so any remaining players see it coming. Nitrox is exempt since its player count can't be queried. This keeps resource usage flat when servers are not actively in use.

Status API

Hub API + Public Status Site

A Flask app running on the same LXC exposes GET /status — a JSON array of all 12 servers with state, player count, and max players. This feeds a custom static status site (auto-refreshing every 30 seconds, with filter tabs for Minecraft vs Steam) and 12 Uptime Kuma keyword monitors provisioned via the SQLite API. The site deploys via git push to a self-hosted Forgejo repo with a deploy timer that rsyncs to the webroot every 60 seconds.

Architecture

LXC 104 — firstmoon

Pelican Panel

Admin UI for managing game servers, user permissions, and port allocations. Communicates with the Wings daemon over HTTPS to start, stop, and provision server containers.

LXC 121 — firstmoon

Wings Daemon

Runs all 12 game servers as isolated Docker containers. Exposes a local API used by Pelican Panel and the Discord bot for power actions. Each container gets its own resource limits and network namespace.

LXC 122 — thirdmoon

Discord Bot + Hub API

Python Discord bot and Flask status API co-deployed as systemd services. Queries all 12 servers every 60 seconds, updates the Discord embed (grouped by Minecraft / Steam), enforces auto-stop, and serves the JSON status endpoint consumed by the public status site and Uptime Kuma.

Employer-Relevant Skills

This project covers more than hosting game servers. I implemented multiple query protocols from scratch (RCON, Steam A2S), built an auto-stop system against a REST API, wrote a status API consumed by two separate clients, and deployed everything as systemd services on dedicated infrastructure. It demonstrates protocol-level networking, automation, API design, and end-to-end service ownership.

What Wasn't Straightforward

The hard parts were protocol diversity, auto-stop edge cases, and scaling from 6 to 12 servers cleanly. Each game server speaks a different protocol — RCON for Minecraft, Steam A2S UDP for Abiotic Factor (including the challenge handshake newer servers require), REST for Terraria — and several servers have no query protocol at all. Writing the RCON client from scratch requires getting the binary packet framing exactly right; the server just drops the connection if the struct packing is off. The auto-stop logic needed careful handling to avoid stopping a server mid-session: servers with no player API return None (not 0), which cleanly opts them out of the idle timer without any special-casing in the watcher loop. The Discord embed also needed restructuring once the server count grew — grouping by Minecraft vs Steam with per-category counts made it actually readable at 12 entries.
